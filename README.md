# 🌸 CarePulse — Maternal Health Monitoring System
### README & File Reference Guide

---

## What is CarePulse?

CarePulse is a smart maternal health monitoring system that:
- Takes vital readings from a pregnant woman (BP, heart rate, temperature, SpO2)
- Uses Machine Learning to predict risk level (Low / Mid / High)
- Combines ML prediction with SpO2 oxygen check for a final alert
- Displays everything on a soft, mobile-friendly dashboard
- Will eventually connect to ESP32 hardware sensors and Firebase cloud

---

## 📁 File-by-File Explanation

---

### 1. `Maternal Health Risk Data Set.csv`
**What it is:** The original dataset downloaded from Kaggle.
**What's in it:** 1014 rows of real patient data with columns:
- Age, SystolicBP, DiastolicBP, BS (blood sugar), BodyTemp, HeartRate, RiskLevel

**When you use it:** Only used during training. Don't modify it.

**Download from:**
https://www.kaggle.com/datasets/csafrit2/maternal-health-risk-data

---

### 2. `build_dataset.py`
**What it does:** Takes the Kaggle CSV and adds a synthetic SpO2 column using medically accurate rules.

**Why we made it:** The original dataset doesn't have SpO2, but our MAX30102 sensor measures it.

**Output:** `maternal_health_sensor_dataset.csv`

**Run:** `py -3.10 build_dataset.py`

> ⚠️ Note: SpO2 values here are synthetic (rule-generated), so we do NOT use them for ML training. SpO2 is checked separately using clinical threshold rules in the API.

---

### 3. `maternal_health_sensor_dataset.csv`
**What it is:** The enriched dataset generated by `build_dataset.py`.

**What's added:** SpO2 column alongside the original 6 features.

**Status:** Generated file — don't need to touch it manually.

---

### 4. `train_model.py`
**What it does:** The most important script. Trains the ML model.

**Features it trains on (5 — matching your sensors):**
| Feature     | Your Sensor     |
|-------------|-----------------|
| Age         | Manual entry    |
| SystolicBP  | BP sensor       |
| DiastolicBP | BP sensor       |
| HeartRate   | MAX30102        |
| BodyTemp    | DS18B20 (°C)   |

**NOT included in training:**
- BS (Blood Sugar) — no glucometer in hardware
- SpO2 — handled separately via threshold rules

**Algorithm:** Random Forest (200 trees) — more accurate than a single Decision Tree

**Output:** `risk_model.pkl`, `label_encoder.pkl`, `feature_columns.pkl`

**Run ONCE:** `py -3.10 train_model.py`

Re-run only if you change the dataset or features.

---

### 5. `risk_model.pkl`
**What it is:** The trained ML model saved to disk.

**Think of it as:** A doctor's brain saved as a file — it has learned patterns from 1000+ real patient records.

**Used by:** `predict_api.py` loads this every time it starts.

**Don't open or edit this file.** It's a binary file.

---

### 6. `label_encoder.pkl`
**What it is:** Saves the mapping between numbers and risk labels.

Example:
- 0 → high risk
- 1 → low risk
- 2 → mid risk

**Used by:** `predict_api.py` to convert model output numbers back to human-readable labels.

---

### 7. `feature_columns.pkl`
**What it is:** Saves the exact list and order of features the model was trained on.

**Used by:** Ensures the API sends data to the model in the exact same column order as training.

---

### 8. `predict_api.py`
**What it does:** The middleman between everything. A Flask web server that:
1. Loads `risk_model.pkl`
2. Waits for incoming sensor data
3. Runs ML prediction on the 5 features
4. Separately checks SpO2 using clinical threshold rules
5. Combines both results into a final alert
6. Returns everything as JSON

**SpO2 threshold logic:**
| SpO2 Value | Alert    |
|------------|----------|
| ≥ 95%      | NORMAL   |
| 90–94%     | WARNING  |
| < 90%      | CRITICAL |

**Key rule:** SpO2 can only ESCALATE the final alert, never reduce it.

**Endpoints:**
- `GET /` → health check (is API running?)
- `GET /test` → quick test prediction in browser
- `POST /predict` → main endpoint (used by dashboard and ESP32)

**Run:** `py -3.10 predict_api.py`
Keep this running at all times while using the dashboard.

---

### 9. `test_api.py`
**What it does:** A test script that sends 4 fake patients to the API and prints predictions.

**Only for development/testing.** Not part of the final system.

**Requires:** `predict_api.py` to be running in a separate terminal first.

**Run:** `py -3.10 test_api.py` (in a NEW terminal)

**Test cases included:**
1. Healthy young mother → expects LOW RISK
2. Moderate concern → expects MID RISK / WARNING
3. High risk patient → expects CRITICAL
4. Normal vitals but low SpO2 → tests SpO2 escalation logic

---

### 10. `dashboard.html`
**What it does:** The user-facing web dashboard. Open this in any browser.

**Features:**
- Mobile-friendly (works on phone and laptop)
- Soft maternal pink/peach theme
- Enter 5 vitals + optional SpO2
- Click "Analyse Vitals" → calls the API → shows result
- Green / Orange / Red alert banner based on risk
- Shows ML result, SpO2 check, and combined final alert separately
- Risk probability breakdown bars
- Session history log with timestamps
- API online/offline indicator

**How to use:**
1. Start `predict_api.py` in terminal
2. Double-click `dashboard.html` to open in browser
3. Enter patient vitals
4. Click Analyse

**Does NOT need internet** — works fully offline as long as the API is running locally.

## 🔄 How Everything Connects

```
Kaggle CSV
    ↓
build_dataset.py → maternal_health_sensor_dataset.csv
    ↓
train_model.py → risk_model.pkl + label_encoder.pkl
    ↓
predict_api.py (Flask server running on port 5000)
    ↑               ↑
dashboard.html    ESP32 hardware (future)
(browser)         (sends sensor readings via WiFi)
```
